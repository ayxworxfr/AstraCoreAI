# AstraCore 工具循环踩坑记录

> 记录开发过程中在 Agent 工具调用链路上遇到的问题、根因及修复方案，供后续维护参考。

---

## 1. Shell 命令失败不报错

**现象**：执行出错的 shell 命令（退出码非 0）返回 `isError=False`，LLM 以为成功，继续往下走。

**根因**：`shell_server.py` 工具处理函数直接 `return` 输出，未区分成功与失败。

**修复**：退出码非 0 时抛出 `RuntimeError`，fastmcp 会自动将响应标记为 `isError=True`。

```python
if exit_code != 0:
    raise RuntimeError(output_line)
return output_line
```

---

## 2. 出错后对话上下文丢失

**现象**：请求处理中途报错后，用户发送的消息和 AI 的部分回复不会被保存，下次对话"失忆"。

**根因**：`ChatUseCase.execute` / `execute_stream` 只在成功路径保存会话。

**修复**：用 `try...finally` 确保无论成功还是异常都会调用 `_save_session`；流式路径用 `_saved` 标志位避免重复保存。

---

## 3. `tool_use` 没有对应 `tool_result` 导致 API 400

**现象**：发生两类场景时，API 报错 `tool_use ids were found without tool_result blocks`。

### 3a. 工具执行异常中断

**根因**：工具执行中途抛异常，session 以 `ASSISTANT(tool_calls)` 结尾，没有紧跟的 `TOOL(tool_results)`。Anthropic API 不允许这种不完整序列。

**修复**：新增 `_strip_dangling_tool_calls`，在所有保存操作之前清理尾部悬空的 `tool_use`：

```python
def _strip_dangling_tool_calls(messages):
    msgs = list(messages)
    while msgs and msgs[-1].role == ASSISTANT and msgs[-1].tool_calls:
        msgs.pop()
    return msgs
```

### 3b. 达到迭代上限时 LLM 仍调用工具

**根因**：最后一轮仍传入工具定义，LLM 产生 `tool_use` 块，但循环 `break` 后工具未执行，导致悬空。

**修复**：最后一轮传 `tools=None`，强制 LLM 只输出文本：

```python
is_last = (iterations == self.max_iterations)
tools_for_llm = None if is_last else tool_definitions
```

## 4. 流式超时机制误杀长时间 MCP 工具

**现象**：搜索整个项目目录（`search_files D:\project **/*`）耗时超过 180 s，被流式空闲超时中断。

**根因**：原有 `_iterate_with_idle_timeout` 在相邻两个 SSE 事件间隔超过阈值时取消请求，MCP 工具长时间无输出会触发此机制。

**修复**：移除流式空闲超时；仅保留单次工具调用的 `asyncio.wait_for` 超时（可配置，默认 60 s）。超时时返回错误 `ToolResult`，不中断整个循环：

```python
try:
    exec_result = await asyncio.wait_for(self.tools.execute(...), timeout=self.tool_timeout_s)
except asyncio.TimeoutError:
    tool_results.append(ToolResult(..., is_error=True, content="[超时]..."))
    continue
```

## 5. 工具循环无最终回答（空响应）

**现象**：工具循环正常结束，但用户看到"（空响应）"，没有实质性答案。

### 5a. 循环结束但末尾没有文字回复

**根因**：工具轮结束时最后一条消息是 `TOOL` 或空 `ASSISTANT`，LLM 没有机会输出最终答案。

**修复**：检测此状态并触发一次无工具的"总结收尾"调用：

```python
def _needs_summary_fallback(messages):
    last = visible_messages[-1]
    return last.role == TOOL or (last.role == ASSISTANT and not last.content.strip())
```

### 5b. 总结收尾本身也产生空输出（上下文溢出）

**根因**：多轮工具调用积累了大量结果，总结收尾时传入的消息超出模型上下文窗口，LLM 无法生成有效回复。

**修复**：总结收尾前裁剪历史，只保留最近 N 轮工具结果（`_trim_for_summary`）。

---

## 6. 工具循环上下文无限膨胀

**现象**：3 轮工具调用后产生空回复。调查发现单轮就包含 directory_tree×5 + read_multiple_files×4 = 9 个工具调用，每条结果最多 20 K 字符，单轮 tool 结果高达 ~108 K 字符，超出上下文窗口。

**根因**：`max_tool_result_chars=20000` 默认值过大；多工具一轮时单轮 token 量已足以撑爆上下文。

**修复**：将单条截断上限降至 6000，强迫 AI 用 `offset/page` 参数分页读取：

```env
ASTRACORE__AGENT__MAX_TOOL_RESULT_CHARS=6000
```

> **注**：曾尝试加入 `context_window_rounds`（限制传给 LLM 的历史轮数）作为第二道防线，但该方案被回滚——仅靠降低单条截断上限已足够，避免引入额外复杂度。

---

## 7. 中间轮旁白文字被当作最终回答

**现象**：AI 在每轮工具调用前输出的规划文字（"先探索目录结构…"）以 `message` 事件推给前端，用户误以为这就是最终答案。

**根因**：流式路径直接将所有 `TEXT_DELTA` 都作为 `message` 事件发出，无论当前轮是否有工具调用。

**修复**：在 `chat_stream` 中缓冲 TEXT_DELTA，遇到 `TOOL_CALL` 才确认该轮是工具轮——将缓存内容转为 `thinking` 事件；循环结束后缓冲中的剩余内容（来自最终无工具轮）才作为 `message` 发出。

```
遇到 TOOL_CALL → 将缓存发为 thinking，后续 TEXT_DELTA 也发为 thinking
循环结束 → 缓存中的内容发为 message（最终答案）
```

## 8. 空响应无任何提示

**现象**：总结收尾也未产生有效内容时，前端仅显示"（空响应）"，用户不知如何处理。

**根因**：会话状态已保存（包含所有工具轮历史），但没有给用户提供任何续接提示。

**修复**：当总结收尾输出为空时，由后端注入一条引导消息并保存入会话：

```python
if not summary_text.strip():
    hint = "信息量较大，本轮分析已暂停。会话已保存，请发送「继续」让 AI 继续完成分析。"
    session.add_message(Message(role=ASSISTANT, content=hint))
    yield {"event": "message", "data": hint}
```

用户发送"继续"后，后端以同一 `session_id` 加载历史，AI 可接续上次结果继续分析。
