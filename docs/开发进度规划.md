# 开发进度规划（产品化执行版）

## 1. 项目目标

### 1.1 背景

- 当前仓库用于临时孵化 AstraCore AI，后续将迁移到独立仓库。
- M1-M4 已完成框架骨架与基础能力建设，M5 后端优化与测试体系已全面推进。

### 1.2 总目标

将 AstraCore AI 从"可演示框架"升级为"专业、最好用、质量最高"的产品级 AI 基础设施，具备以下属性：

- 对开发者：接口稳定、文档清晰、接入成本低。
- 对生产环境：可观测、可回滚、可排障、可治理。
- 对业务团队：高可靠、可配置、可扩展、可验证。

### 1.3 交付目标

- 交付 Beta（6-8 周）：关键链路稳定可用，可对内试点接入。
- 交付 1.0（4-6 个月）：工程质量和运行保障达到对外发布标准。

### 1.4 验收标准（总）

- 质量门禁、里程碑、任务包、风险、发布机制全部可追踪。
- 所有高优先级任务均有明确"完成判定"，拒绝模糊完成。
- 文档可直接驱动执行，不依赖额外口头解释。

## 5. 质量门禁（Definition of Done）

### 5.1 代码质量门禁

- 所有新增功能必须附带测试，不接受"仅手工验证"。
- `ruff check` 与 `mypy` 必须全量通过（当前 ruff: 0 errors ✅）。
- 新增或重构代码不得引入未解释的复杂分支。

### 5.2 测试门禁

- P0 关键路径必须具备单元测试 + 集成测试。
- 回归测试必须覆盖：对话、工具调用、RAG 检索、记忆读写、工作流恢复。
- 每次发布前必须执行完整回归集。
- 当前状态：**120 个单元测试 ✅**，集成测试待建立。

### 5.3 可靠性门禁

- Provider 异常、超时、限流、工具失败必须有可验证的降级或恢复路径。
- 关键 API 必须定义超时上限与重试策略（当前 tenacity retry + asyncio.wait_for ✅）。
- 关键错误必须可追踪到具体请求与上下文。

### 5.4 安全门禁

- 默认配置符合最小暴露原则，禁止宽松生产默认值（CORS 白名单 ✅，不再 allow_origins=["*"]）。
- 敏感字段必须脱敏记录，不允许明文泄露（sanitize_metadata ✅）。
- 审计日志覆盖关键操作（请求、工具执行、策略命中、错误）。

### 5.5 发布门禁

- 发布前完成发布清单签核（功能、测试、性能、安全、文档）。
- 必须存在可执行回滚方案并完成至少一次演练。
- 发布后 24 小时内完成稳定性复盘。

## 7. 项目整体技术分析

### 7.1 已夯实的优势

**架构层面：**
- Clean Architecture 边界清晰，Domain 零外部依赖，单元测试可完全 mock 外部 I/O。
- Ports & Adapters 使替换 LLM Provider、向量库、存储层的成本接近零。
- Pydantic v2 + StrEnum + UTC datetime 全栈统一，数据模型可靠。
- 多模型 Profile 注册表已落地，模型连接信息、密钥引用、MCP server 与运行配置集中在 `config/`。

**功能层面：**
- 完整的 Chat → Tool Loop → RAG → Memory 四条核心链路均已实现并测试。
- Anthropic 流式 tool args 正确累积，不再出现空 `{}` 的 bug。
- 模型能力由内置 registry 推导，可自动处理 thinking、tools、temperature、Anthropic block 回放差异。
- 工作流 checkpoint 持久化，支持 Redis 持久 + 内存降级双保险。
- 记忆分层：Redis 短期（含 TTL 淘汰 + 容量上限）+ PostgreSQL 长期。

**工程层面：**
- ruff 零错误，`pytest` 120 tests 全绿，mypy strict 通过。
- 安全：CORS 白名单、输入 XSS 检测、敏感字段脱敏、工具执行白名单。

### 7.2 当前薄弱点（按影响程度排序）

| 薄弱点 | 具体问题 | 建议方向 |
|--------|----------|----------|
| **集成测试缺失** | 单元测试 mock 了所有 I/O；真实 Redis/PG/ChromaDB 路径从未被测试 | testcontainers + docker-compose，建立集成测试套件 |
| **鉴权完全缺失** | Service 层无 API Key / JWT 验证，所有端点公开可访问 | 添加 `APIKeyMiddleware` 或 `Depends(verify_api_key)` |
| **熔断器缺失** | 有 retry/timeout，但连续失败无熔断，会持续产生超时等待 | 实现 CLOSED/OPEN/HALF_OPEN 状态机，集成到 PolicyEngine |
| **可观测性未闭合** | `ObservabilityLogger`、`MetricsCollector` 端口已有但未接入真实后端 | 接入 structlog + Prometheus metrics |
| **RAG 检索质量** | `search_memory` 使用 ILIKE 全文搜索，生产中需 pg_trgm 或向量检索 | 升级为 pgvector 或保留 ILIKE 但加 GIN 索引 |
| **前端状态管理** | 前端使用 Zustand，后台 Chat Run 已支持刷新恢复；仍可补充全局错误边界和更完整的重试入口 | 补充错误边界组件、失败 run 重试入口 |

## 10. 进展日志（倒序）

### 2026-04-30（M5 SDK/Service 代码去重 + SDK 全功能对齐 + SSE done 事件优化）

**ChatOrchestrator 共享执行引擎（refactor: 提取 ChatOrchestrator 消除 SDK/Service 重复逻辑）：**
- 问题：`service/api/chat.py` 与 `sdk/client.py` 存在 300+ 行近乎一致的重复逻辑（system prompt 组装、LLM 路由、工具循环、summary fallback、session 持久化）
- 方案：新建 `service/chat_orchestrator.py`，提取 `ChatOrchestrator` 作为两端共享的 chat 执行引擎
- `service/prompt_utils.py`：将 prompt 工具函数（时区信息、Skill 提示渲染）从 `chat.py` 抽出独立模块
