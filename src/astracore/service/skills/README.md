# Skills 设计文档

本目录存放 Astra AI 的 skill 配置。每个 skill 是一份**完整的人格 + 工作手册**，通过互斥加载的方式在对话中生效。

本文档既是**设计哲学说明**，也是**新增 skill 时的操作模板**。

---

## 一、加载机制

- **互斥加载**：每次对话只有一个 skill 生效，不会叠加
- **order 字段**：数值越小越优先匹配；未命中任何 skill 时走 `default: true` 的 skill
- **description 字段**：用于触发匹配，应概括 skill 的适用场景，不是角色介绍

> ⚠️ 互斥加载意味着**每个 skill 必须自带所有必要的基础规则**（人格、沟通方式、格式规范等），不能假设"通用助手另外生效"。

## 三、核心设计哲学

### 1. 每个规则都必须可验证

**反面**：`要专业`、`要严谨`、`合理使用`
**正面**：`超过 5 个类别不用饼图`、`默认压缩至原文 1/5`、`连续 2 次失败必须跑 doctor`

空话模型没法执行，**规则必须落到具体阈值、具体动作、具体清单**。

### 2. 正负清单对列

模型的常见失败不是"没做该做的"，而是"做了不该做的"。所以每个场景**既要写该做什么，也要写不该做什么**。

示例（写作助手 · 润色）：
```
- **改**: 病句、歧义、冗余表达、错别字、逻辑不通
- **不改**: 个人用词偏好、句式习惯、观点倾向、有意为之的语气词
```

### 3. 场景化操作，不堆抽象原则

与其写 `擅长各类写作场景`，不如把场景拆开，每个场景给独立的操作流程。参考写作助手的"五大场景"、代码助手的"六种工作场景"、故事大师的"七种叙事模式"。

### 4. 冲突处理显性化

当 skill 内规则有局部冲突时，明确说明**哪条优先、什么情况下例外**。

示例：通用助手说"能一步到位就不分步确认"，但数据分析师和写作助手说"需求模糊时先反问"——这是针对"猜错成本高"场景的主动覆盖，不是遗漏。

### 5. 失败路径和成功路径一样重要

绝大多数 prompt 只规定"成功时怎么做"，但真实场景一半以上是异常。每个 skill 应有：
- **失败兜底**：命令报错、超时、结果不符预期时怎么办
- **边界声明**：什么时候不该出手（⚠️ 标记）
- **降级策略**：信息不足时的处理方式（反问 / 列选项 / 给默认）

## 四、标准结构模板

推荐按以下骨架组织 skill，可按需增删小节：

```markdown
---
name: 名称
description: 一句话说清触发场景
order: 数字
default: true   # 仅通用助手用
---
# 角色

一段话说清楚：你是谁、擅长什么、服务于谁。
必要时声明默认前提（如 OpenClaw 管理员开头的"默认原生 CLI"）。

---

## 当前时间

{{current_time_info}}

---

## 核心原则 / 工作原则

最重要的 5-8 条规则，每条一个要点，关键词加粗。
这一节是 skill 的"宪法"，其他章节都是对这些原则的具体化。

---

## 场景化操作 / 工作场景

按子场景拆开，每个场景给出：
- 触发条件（如适用）
- 具体操作步骤或正负清单
- 输出格式要求

这是 skill 的**主体价值区**，也是区别于通用助手的地方。

---

## 特殊情况 / 失败兜底

- 信息不足怎么办
- 遇到专业领域怎么办
- 结果异常怎么办
- 边界外的请求怎么办（⚠️ 声明）

---

## 格式规范 / 输出结构（可选）

- 默认长度、排版规则
- 特殊符号约定（如翻译官的 `※`、通用的 ⚠️）
- 什么时候分段、什么时候给表格
```

## 五、新增 skill 检查清单

写完新 skill 后对照检查，全部打勾再合入：

### 基础一致性
- [ ] 开头声明了角色、服务对象、默认前提
- [ ] 包含 `{{current_time_info}}` 占位符
- [ ] `{{ai_name}}` / `{{owner_name}}` 使用正确（需要时）
- [ ] 不加废话前缀（"好的，让我来…"）
- [ ] 不瞎编，不确定明说

### 规则质量
- [ ] 每条规则都可验证（有具体阈值 / 清单 / 动作）
- [ ] 主要场景有**正负清单对列**（改/不改、用/不用）
- [ ] 有至少一条**负面声明**（不做什么、什么时候不出手）
- [ ] 和其他 skill 有局部冲突时，显性说明适用边界

### 场景覆盖
- [ ] 主要场景拆开列出，不是一锅炖
- [ ] 每个场景有具体操作步骤或输出格式
- [ ] 覆盖了"信息不足"的降级策略
- [ ] 覆盖了"失败 / 异常"的兜底流程

### 符号和格式
- [ ] 重要警告用 ⚠️
- [ ] 关键词用粗体
- [ ] 列表 > 长段落，表格 > 大段对比
- [ ] 子场景之间用 `---` 或三级标题分隔

## 六、维护节奏

- **不要纸面博弈**：skill 的优化收益在实战 2 周后急剧上升，此前的反复打磨边际递减
- **基于真实踩坑迭代**：遇到 skill 表现不符预期时，先记录场景和期望输出，积累到 3-5 条再一起改
- **保持家族气质**：新增 skill 风格应和现有 skill 一致（具体化规则、正负清单、失败兜底），不破坏统一性
- **order 变更谨慎**：调整 order 可能改变触发优先级，影响其他 skill 的命中率

---

## 七、符号约定

统一使用以下符号，避免各 skill 自造体系：

| 符号 | 用途 |
|---|---|
| ⚠️ | 重要警告 / 边界声明 |
| `※` | 翻译官专用：文化表达注释、关键歧义说明 |
| **粗体** | 关键词、关键动作 |
| `—` | 命令/选项 + 一句话用途的分隔符 |
| `→` | 流程推进（如"症状 → 根因 → 修复"） |

新 skill 需要自造符号时，先评估是否能复用上述体系。
